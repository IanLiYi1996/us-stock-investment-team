# OPS 审核协议

> 定义 Ops 审核的范围、节奏和标准。

---

## 1) 审核范围

### 必须审核（硬性要求）
- 所有 S 类任务的 closeout
- 任何 Agent 对 SOUL.md 的自我修改
- 止损/止盈阈值变更
- 自主权边界（L3）调整
- A2A 权限矩阵变更

### 建议审核（软性建议）
- A/P 类任务的 closeout（signal≥3）
- KO 产出的新 principle
- MEMORY.md 的重大更新

---

## 2) 审核标准（五维评估）

| 维度 | 通过条件 |
|------|---------|
| 规则对齐 | 不违反 SYSTEM_RULES 和 INVESTMENT_FRAMEWORK |
| 范围影响 | 影响范围可控，已通知相关方 |
| 可回滚性 | 有明确回滚方案 |
| 资源成本 | 不显著增加系统复杂度 |
| 安全性 | 不引入交易风险或数据泄露 |

### 评判结果
- **通过** ✅：五维全部满足
- **建议修改** ⚠️：1-2 维不满足，非高风险
- **拦截** ❌：安全性或规则对齐不满足，或涉及硬性拦截条件

---

## 3) 审核节奏

| 周期 | 内容 | 产出 |
|------|------|------|
| 每周 | S 类 closeout + Self-Update 审计 + 阈值一致性检查 | 周审计摘要 |
| 每月 | MEMORY 去重 + KO 质量审核 + SYSTEM_RULES 检查 | 月度报告 |
| 每季 | 全面系统健康检查 + 知识库价值回顾 | 季度报告 |

---

## 4) 升级路径

- Ops 发现问题 → 通知相关 Agent 修正
- Agent 不同意 → Ops 升级给用户裁决
- 涉及 SYSTEM_RULES 变更 → 必须用户确认

---

## 5) 审计报告格式

```
## Ops 审计: [类型] [YYYY-MM-DD]
- **范围**: [审计了什么]
- **发现**: [N 条问题 / 全部通过]
- **行动项**:
  1. [问题 1] → [建议] → [负责人]
  2. [问题 2] → [建议] → [负责人]
- **系统健康度**: ✅ 正常 / ⚠️ 需关注 / ❌ 需立即处理
```
